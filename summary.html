Socratic Tutor Evaluation Framework

A quantitative, knowledge-graph-grounded framework for evaluating Socratic tutoring quality — built on the Junyi Academy dataset and Bayesian Knowledge Tracing.

Overview

This framework evaluates the Socratic tutor plugin by simulating a cooperative student agent whose knowledge is grounded in a real educational dataset, running a tutoring session, and scoring the tutor's behaviour against five quantitative metrics.

💡
The evaluation is process-based: it measures the quality of the tutor's questioning relative to the student's current knowledge state, rather than measuring student learning outcomes directly. This avoids dependence on a realistic student simulation while still producing meaningful signals about tutoring quality.

The key insight motivating this design is that an effective Socratic teacher should guide any student in a particular direction. Success is measured by whether the teacher's questions consistently target the student's knowledge frontier — not by whether a particular student model happens to learn well.

Data Foundation — Junyi Academy Knowledge Graph

Source: Junyi Academy, a Taiwanese math e-learning platform derived from Khan Academy, providing 41 knowledge components (KCs) with prerequisite relationships derived from expert annotation.

The raw annotation data contained 19 bidirectional edge pairs (cycles of length 2) that were resolved by removing pedagogically backwards edges, with 2 further back-edges removed by DFS cycle detection. The resulting graph stored in data/junyi_kg.json is a verified DAG.

The Four Agents

Existing Domain Mapper

Given source material (a topic string), produces a structured JSON knowledge graph with concept nodes, prerequisite edges, common misconceptions, checkpoint questions, and a recommended teaching sequence. Output is used by the teacher — not shared with the Student Evaluator.

Existing Socratic Teacher

The plugin itself (/socratic-tutor:socratic), unchanged. Receives the domain KG and conducts the session by asking Socratic questions, never stating direct answers. Two hooks govern behaviour: UserPromptSubmit (filters adversarial messages) and Stop (checks each outbound message for answer compliance, returning {"ok": true/false}).

New Student Agent

Replaces the old adversarial agent. Receives a Student Knowledge Document — a prose description of its knowledge state — and a behavioural contract framing its role as accurately representing a specific knowledge profile in a scientific experiment. Appends a private self-assessment JSON after each response, flagging any knowledge leakage.

New Student Evaluator

Observes the dialogue turn-by-turn. Receives the KC graph structure (topology only — no domain mapper reasoning or checkpoint questions). Uses Bayesian Knowledge Tracing to maintain a per-KC mastery probability vector. After each student turn, classifies the response into one of six observation classes and updates BKT estimates.

📌
Separation principle: The Domain Mapper and Student Evaluator must not share reasoning. The Evaluator receives KC names and edges but not the Mapper's assessment of what mastery looks like — ensuring mastery is inferred from observed behaviour alone, not inherited from domain priors.

Bayesian Knowledge Tracing

BKT models the probability that each KC is mastered, updated after each student turn using the standard 4-parameter model (Corbett & Anderson, 1994).

Parameters

ParameterValueMeaning
P(L₀) absent 0.10Prior mastery probability for unstudied KCs
P(L₀) partial 0.50Prior for partially understood KCs
P(L₀) mastered0.90Prior for pre-mastered KCs
P(T) 0.10Learning transition probability per opportunity
P(G) 0.25Guess rate — correct response despite no mastery
P(S) 0.10Slip rate — incorrect response despite mastery

Observation Classes

strong_articulation Student explains the KC correctly and unprompted — strongest evidence of mastery
weak_articulation Correct but minimal or fragmented response — moderate evidence
guided_recognition Correct only after visible scaffolding in the teacher's question — weak evidence
misconception Student states a demonstrably wrong belief — negative evidence
absent KC not engaged this turn — no update (tiny drift only)
contradiction Student contradicts a prior response on the same KC — destabilises estimate

Knowledge Frontier

The knowledge frontier is the set of target KCs whose direct prerequisites are all mastered (p > 0.70) but which are not yet firmly mastered themselves (p < 0.70). This is the ideal target zone for the tutor's questions at any given turn.

NAC — Non-Answer Compliance Rate

NAC Non-Answer Compliance Rate
NAC = turns where Stop hook returned ok:true / total tutor turns with a recorded verdict

Captured from the Stop hook's {"ok": true/false} output logged from stderr during the session. Falls back to a heuristic (trailing ? in the last 100 characters) when the hook verdict is absent in older log formats.

RangeInterpretation
≥ 0.97Excellent — at most 1 violation per 30-turn session
0.90 – 0.97Acceptable
< 0.90Failing — systematic answer leakage

KFT — Knowledge Frontier Targeting Score

KFT Knowledge Frontier Targeting Score
KFT = mean of per-turn scores across all teacher turns

For each teacher question turn, an LLM classifies which KC the question targets and compares it to the Student Evaluator's frontier at that moment. A high KFT means the teacher consistently targets the student's current learning boundary rather than drilling already-mastered concepts or jumping ahead of prerequisites.

SituationTurn score
Targeted KC is on the frontier1.0
Targeted KC already mastered (p > 0.70)0.3
Targeted KC's prerequisites not yet met0.0

Off-map (no tangent logged)0.0

TBA — Teacher Belief Accuracy

TBA Teacher Belief Accuracy
TBA = 1 − mean(|teacher_phase − evaluator_phase| / 8) across all turns

Measures how well the teacher's internal model of the student (from .socratic-session.json) aligns with the Student Evaluator's independent mastery estimate. teacher_phase is the session log's current_phase field (1–8); evaluator_phase is derived from the highest-indexed target KC with p > 0.70 in the BKT state. A low TBA often explains poor KFT — the teacher is targeting the wrong region because its student model is miscalibrated.

MRQ — Misconception Response Quality

MRQ Misconception Response Quality
MRQ = targeted_probe_turns / total_misconception_turns

For turns where the Student Agent's self-assessment flags misconception_activated: true, checks whether the teacher's next 1–3 turns target the KC associated with that misconception (via LLM classification). Returns None if no misconceptions were activated during the session (e.g. tabula_rasa profile) — the composite weight is redistributed to NAC and KFT proportionally in that case.

⚠️
MRQ allows a 6-turn response window. The accuracy-reviewer subagent runs every 6 turns, so a misconception expressed on turn 3 may not be addressed until turns 7–9. Same-turn reaction is not required.

RS — Robustness Score

RS Robustness Score
RS = 0.6 × (1 − CV_KFT) + 0.4 × (1 − CV_NAC) where CV = std / mean across student configurations

A meta-metric computed across multiple runs of the same topic with different student profiles. A high RS means the teacher's guidance quality is consistent regardless of which student configuration it faces — evidence the behaviour is genuinely adaptive rather than accidentally aligned with one student type.

NAC and KFT carry equal and highest weight as the two core behaviours: not giving answers, and targeting the right concepts. TBA and MRQ are secondary signals. RS is a meta-metric given lower individual-run weight since it requires multiple sessions to compute.

Running Tests

⚠️
The orchestrator calls claude -p internally. Run it from a plain terminal, not from inside a Claude Code session — nested sessions are blocked.

Single session

cd /path/to/test
python3 orchestrator.py \
  --mode student \
  --student-config tabula_rasa \
  --turns 8 \
  --topic "factors and fractions"

Score a completed run

python3 scorer.py runs/<timestamp>_student_tabula_rasa.jsonl --metrics --turns

Robustness score across all three profiles

python3 scorer.py --compare \
  runs/<ts>_student_tabula_rasa.jsonl \
  runs/<ts>_student_partial_knowledge.jsonl \
  runs/<ts>_student_misconception_heavy.jsonl

Visualise a session

Open visualize.html in a browser. The current file is hardcoded to the tabula_rasa run from 2026-02-25. Use the slider to step through turns; hover observation badges for evidence quotes.

Known Limitations

  1. Student simulation validity. The student agent represents a knowledge state accurately but does not perfectly model human confusion. Weaker models (Haiku) exhibit genuine knowledge limits; stronger models may reason too fluently on topics outside their Knowledge Document.
  2. Stop hook verdict capture. The hook verdict is parsed from stderr via regex. If the CLI output format changes, stop_hook_verdict will silently fall back to null and NAC will revert to the trailing-question-mark heuristic.
  3. KFT and MRQ require LLM calls per turn. Both metrics are computed post-hoc by scorer.py --metrics and each involves a separate Claude CLI invocation, making scoring slow for long sessions.
  4. Visualisation is static. visualize.html contains hardcoded data from a single run. A future improvement would be a script that auto-generates it from any .jsonl run file.
  5. Scope limited to Junyi's math KCs. The framework evaluates tutoring only on topics covered by the Junyi prerequisite graph. CS or other domains require a supplementary dataset or a synthetic KC graph generated by the domain mapper.
